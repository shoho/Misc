# Frontend Design — 実装ガイド

小さなスタジオのデザインリードとして、案件ごとに固有のビジュアルアイデンティティを与える。以下は `/design` 系ワークフローの実装フェーズで従う実務詳細。**迷ったらここの数値と例に従う。**

## 1. 題材からデザインを導く

独自性は題材の世界(素材・道具・語彙)から生まれる。コーヒーの LP なら焙煎度の色階調、財務ダッシュボードなら帳票の罫線文化、といったように、題材固有の手がかりをパレットとタイポグラフィに翻訳する。「どんな題材にも使える見た目」になっていたら、それは設計をサボったサイン。**最低1つ、題材からしか出てこない視覚的判断を入れる。**

## 2. タイポグラフィ(数値基準)

- 見出し用(個性がある。使用は控えめに)+本文用(可読性重視)+必要なら数字/コード用の3役でペアリングする。
- タイプスケールは倍率 1.2〜1.333 を決めて全サイズを算出する。例(1.25 / 基準 16px):

  | トークン | サイズ | 用途 |
  |---|---|---|
  | --text-xs | 11px | 出典・凡例 |
  | --text-sm | 13px | ラベル・キャプション |
  | --text-base | 16px | 本文 |
  | --text-lg | 20px | リード文・小見出し |
  | --text-xl | 25px | セクション見出し |
  | --text-2xl | 31px | ページ見出し |
  | --text-3xl | 39px | ヒーロー |

- 本文の行長は 65 字前後(`max-width: 65ch` 目安)。行間は本文 1.5〜1.7、見出しは 1.1〜1.3。
- 見出しは `text-wrap: balance`、大文字ラベルは 11〜13px + `letter-spacing: 0.05em`。
- 数字が縦に並ぶ場所(表・統計・タイマー)は `font-variant-numeric: tabular-nums`。
- Web フォントの外部 URL は使わない(オフラインで開けなくなり、サイレントフォールバックの温床になる)。システムフォントスタックを役割別に組むのが基本。どうしても特定書体が必要なら data URI で埋め込む。

## 3. カラー(数値基準)

- パレットは 4〜6 色に絞り、`:root` のカスタムプロパティとして**役割名で**定義する(`--accent` は可、`--blue` は不可)。
- コントラスト:本文/背景 **4.5:1 以上**、24px 以上の見出し **3:1 以上**。両テーマで満たす。
- ニュートラルはアクセント色の色相を 2〜5% 混ぜる。純グレー(R=G=B)は「選んでいない色」に見えるので禁止。
- セマンティックカラー(success / warning / danger)はアクセントとは別枠で確保する。
- 大胆さは1箇所に集中させ、周囲は静かに保つ。アクセントが背景と喧嘩したら、彩度を落とすか類似色相に寄せる(色の総入れ替えはしない)。

## 4. 両テーマの実装パターン

```css
:root {
  --bg: #faf9f7;
  --ink: #1c1a17;
  --accent: #b4552d;
}
@media (prefers-color-scheme: dark) {
  :root {
    --bg: #171512; --ink: #ece8e1; --accent: #d97a4e;
  }
}
:root[data-theme="dark"]  { --bg: #171512; --ink: #ece8e1; --accent: #d97a4e; }
:root[data-theme="light"] { --bg: #faf9f7; --ink: #1c1a17; --accent: #b4552d; }
```

- **4ブロック全部が必要**(OS 追従2 + 手動トグル2)。色トークンを1つ追加・変更したら、4ブロックすべてに反映されているかを確認する。
- コンポーネントは必ずトークン経由で色を参照する。メディアクエリの中にコンポーネントスタイルを書かない。
- ダークは単純反転ではなく再設計する:背景は真っ黒でなく #141414〜#1c1c1c 帯、影は効かないので面の明度差で階層を作る、アクセントは彩度を保って明度 +10〜20%、コントラストを再確認する。
- 意図して単一テーマにする場合(ネオンのアーケード画面、活版の招待状など)は、その選択を成果物の報告に明記する。

## 5. レイアウトと余白

- 兄弟要素の間隔は親の flex/grid + `gap` で管理する。子への margin 連打は相殺・二重取りの温床。
- 余白スケール(4px 基数)をトークン化し、そこからだけ取る。13px / 27px のような恣意的な値を作らない。
- 高さは原則コンテンツに任せる。`height: 100vh` や固定 px 高はテキスト増減ではみ出すので、必要な場合だけ `min-height` を使う。
- 表・コードブロック・図など幅の出るものは、それ自身のコンテナに `overflow-x: auto`。ページ全体を横スクロールさせない。
- 画像は `max-width: 100%`。レスポンシブは相対単位と grid の再配置で実現する。`position: absolute` をレイアウト手段にしない(装飾の重ね置きのみ可)。

## 6. カスケード衝突を作らない

同じ要素の同じプロパティを複数セレクタで奪い合わせない。セレクタは原則クラス1段(詳細度 0,1,0)。

```css
/* ❌ 悪い例: .hero .cta が .button の padding を暗黙に上書きし、
   後から .button を直しても効かない */
.button { padding: 8px 16px; }
.hero .cta { padding: 12px; }

/* ✅ 良い例: バリアントは同じ詳細度のクラスで追加する */
.button { padding: var(--space-2) var(--space-4); }
.button-large { padding: var(--space-3) var(--space-6); }
```

`!important` は使わない(使いたくなったら詳細度設計が壊れているサイン)。

## 7. コピー(文言)

言葉もデザイン素材。lorem ipsum は使わない。

- ユーザー側の言葉で命名する(システム都合の「webhook 設定」ではなく「通知」)。
- 能動態。ボタンは起きることを正確に言う(「公開する」→ トースト「公開しました」)。
- エラーは「何が起きたか+どうすれば直るか」。謝罪と曖昧語は不要。
- 具体は気の利いた言い回しに勝つ。

## 8. 構造は情報

ナンバリング・アイキャッチ・区切り線・ラベルは、内容について真であることを符号化する場合にだけ使う。01/02/03 は本当に順序が情報のとき(手順・年表)だけ。装飾のための構造はノイズ。

## 9. UI(操作するもの)の場合

- 走査されるものとして設計する:要約→詳細の順。注意が必要な状態は数字だけでなく形(ピル、チップ、ストライプ)でも読めるように。
- チャートにも本文と同じ丁寧さを:エリアの淡い塗り、かすかなグリッド、終端の強調。
- 操作できるものは操作できる見た目に(カーソル、hover、focus リング)。

## 10. よくある失敗(先回りで避ける)

過去の生成物で繰り返し起きた不具合。実装中にこのリストを意識し、出荷前チェックで必ず潰す:

1. **ダークテーマの更新漏れ** — ライト側だけ変えて4ブロックの残りを忘れる → 色を触ったら4ブロックを機械的に確認
2. **トークン定義後の生 HEX** — コンポーネント内に `#fff` を直書き → 出荷前に `#` を検索して定義部以外に無いか確認
3. **固定高のはみ出し** — `height` 指定の箱にテキストが収まらない → `min-height` に置き換え
4. **グラデーション上の文字** — 背景の途中でコントラストが切れる → 単色の面に載せるか、テキスト裏に面を敷く
5. **フォントスタックの誤想定** — 1番目のフォントが無い環境で行間・字間が崩れる → フォールバック込みで見た目を確認
6. **長大な手書き SVG パス** — 装飾グラフィックを座標の羅列で書いて破綻 → CSS 図形・Canvas・シンプルな図形の組み合わせにする

## 11. 品質チェック(出荷前)

- [ ] 要素の重なり・はみ出し・ページ横スクロールが無い
- [ ] 本文コントラスト 4.5:1 以上を両テーマで満たす(`data-theme` トグルでも確認)
- [ ] フォーカスが見える/キーボードで操作できる
- [ ] `prefers-reduced-motion` でアニメーションが抑制される
- [ ] セレクタ詳細度の衝突で余白・色が打ち消されていない
- [ ] トークン定義部以外に生の HEX が無い
- [ ] 非 void 要素がすべて閉じ、属性は二重引用符
- [ ] ブラウザのスクリーンショットで最終確認済み

## 12. エディトリアル案件の追加原則

LP・公開ページ・「持ち帰って共有したくなるもの」では:

- **ヒーローは主張**:題材の世界で最も特徴的なもの(見出し、画像、ライブデモ、インタラクティブな瞬間)で開く。
- タイポグラフィ自体を記憶に残る要素にする。どの案件でも出せる書体選定にしない。
- モーションは狙って使う:散発的なエフェクトより、1つのオーケストレーションされた瞬間(ロード時のシーケンス、スクロール連動の出現)が効く。ただし過剰なアニメーションは AI 生成感を強めるので、少ない方が良い場合も多い。
- 複雑さはビジョンに合わせる:マキシマルな方向は作り込みで、ミニマルな方向は余白・タイプ・ディテールの精度で応える。
- 大胆さの予算は1箇所。実装前にプランを題材と突き合わせ、汎用に見える部分を直してから書き始める。
