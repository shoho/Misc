# /design-system — コードベースからデザインシステムを作る

プロジェクトを読み取り、以降のすべての `/design` が自動的に従うデザインシステムを生成する。既存プロダクトが無い場合は、ユーザーのブランドについてヒアリングした内容(または妥当な提案)から新規に作る。

## 手順

1. **既存の手がかりを収集する**
   次を探して読む(存在するものだけ):
   - `tailwind.config.*`、`theme.*`、`tokens.*`、CSS カスタムプロパティ定義、SCSS 変数
   - 既存コンポーネントのスタイル(ボタン・カード・フォームなど代表的なもの)
   - ロゴ・ブランドカラーが分かるアセット、README、デザイン関連ドキュメント
   - `AGENTS.md` / `GEMINI.md` にデザイン規約があればそれも

2. **トークンに正規化する**
   収集結果を次の体系に整理する:
   - カラー:ブランド/アクセント、ニュートラル階調、セマンティック(success/warning/danger)、ライト・ダーク両テーマの値
   - タイポグラフィ:書体ファミリー(役割別)、タイプスケール、行間、字間
   - 余白・角丸・影・ブレークポイント
   矛盾する値が複数見つかった場合は、使用頻度が高い方を正とし、報告に矛盾点を列挙する。

3. **2ファイルを生成する**
   - `designs/tokens.css` — `:root` / `@media (prefers-color-scheme: dark)` / `:root[data-theme="dark"]` / `:root[data-theme="light"]` の4段構成で全トークンを CSS カスタムプロパティとして定義
   - `designs/design-system.md` — トークンの一覧と使い方、書体ペアの意図、コンポーネントの基本形(ボタン・入力・カード)、やってはいけないこと(Do/Don't)
   
4. **プレビューを作って検証する**
   全トークンを一覧できる簡単なスタイルガイドページ `designs/design-system/index.html` を生成し、ブラウザで開いてスクリーンショットで確認する(両テーマ)。

5. **報告する**
   生成したファイル、トークンの出典(どのファイルから抽出したか)、判断が必要だった点をまとめる。以降の `/design` はこのシステムを自動で使うことを伝える。
