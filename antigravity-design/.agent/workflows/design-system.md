# /design-system — コードベースからデザインシステムを作る

プロジェクトを読み取り、以降のすべての `/design` が自動的に従うデザインシステムを生成する。既存プロダクトが無い場合は、ユーザーのブランドについてヒアリングした内容(または妥当な提案)から新規に作る。

## 手順

1. **既存の手がかりを収集する**
   次を探して読む(存在するものだけ):
   - `tailwind.config.*`、`theme.*`、`tokens.*`、CSS カスタムプロパティ定義、SCSS 変数
   - 既存コンポーネントのスタイル(ボタン・カード・フォームなど代表的なもの)
   - ロゴ・ブランドカラーが分かるアセット、README、デザイン関連ドキュメント
   - `AGENTS.md` / `GEMINI.md` にデザイン規約があればそれも

2. **ビジュアル参照(スクリーンショット・スライド)を読み取る**
   `designs/references/` 配下、およびユーザーが指定したパスにある画像・PDF(既存プロダクトのスクリーンショット、Google Slides / Figma からの書き出し、ブランドガイドの PDF など)を開き、視覚的に分析する:
   - カラー:背景・テキスト・アクセント・セマティック色を HEX で推定(スクリーンショットは圧縮やアンチエイリアスで色がぶれるため、複数箇所から読み取って丸める)
   - タイポグラフィ:書体の分類(セリフ/サンセリフ/幾何学系など)と近い実在フォントの候補、ウェイト、おおよそのタイプスケール
   - 形:角丸、影、罫線、余白の密度、コンポーネントの基本形
   画像からの推定値はすべて **推定である旨をマークし**、報告時に仮定として列挙する。Google Slides が Drive 連携(MCP)経由で直接読める環境ならそこから取得してよい。読めない場合は「PNG/PDF に書き出して `designs/references/` に置いてほしい」とユーザーに依頼する。

3. **トークンに正規化する**
   収集結果を次の体系に整理する:
   - カラー:ブランド/アクセント、ニュートラル階調、セマンティック(success/warning/danger)、ライト・ダーク両テーマの値
   - タイポグラフィ:書体ファミリー(役割別)、タイプスケール、行間、字間
   - 余白・角丸・影・ブレークポイント
   矛盾する値が複数見つかった場合は、使用頻度が高い方を正とし、報告に矛盾点を列挙する。コードから抽出した正確な値とビジュアル参照からの推定値が食い違う場合は、**コードの値を正**とする(参照画像はブランドの意図の補完として扱う)。コードが存在せず参照画像だけの場合は、推定値をそのまま採用してよい。

4. **2ファイルを生成する**
   - `designs/tokens.css` — `:root` / `@media (prefers-color-scheme: dark)` / `:root[data-theme="dark"]` / `:root[data-theme="light"]` の4段構成で全トークンを CSS カスタムプロパティとして定義
   - `designs/design-system.md` — トークンの一覧と使い方、書体ペアの意図、コンポーネントの基本形(ボタン・入力・カード)、やってはいけないこと(Do/Don't)
   
5. **プレビューを作って検証する**
   全トークンを一覧できる簡単なスタイルガイドページ `designs/design-system/index.html` を生成し、ブラウザで開いてスクリーンショットで確認する(両テーマ)。ビジュアル参照がある場合は、参照画像とスタイルガイドを見比べて印象が合っているかを確認する。

6. **報告する**
   生成したファイル、トークンの出典(どのファイル/どの画像から抽出したか)、画像由来の推定値とその確度、判断が必要だった点をまとめる。以降の `/design` はこのシステムを自動で使うことを伝える。
