# /design-system — デザインシステムを抽出・生成する

プロジェクトを読み取り、以降のすべての `/design` / `/design-slides` が自動的に従うデザインシステムを生成する。既存プロダクトが無い場合は、ユーザーのブランドについてヒアリングした内容(または妥当な提案)から新規に作る。

成果物は必ずこの3ファイル:

| ファイル | 内容 |
|---|---|
| `designs/tokens.css` | 全トークンの CSS カスタムプロパティ(下のスケルトンの名前を変えずに使う) |
| `designs/design-system.md` | トークン一覧・意図・コンポーネント基本形・Do/Don't(下のテンプレート構成) |
| `designs/design-system/index.html` | 全トークンを一覧するスタイルガイド(検証用) |

## 手順

### 0. モードを確認する

`designs/.design-mode` が存在し `strict` とあれば、`.agent/rules/design-strict.md` を読んでその制約を逸脱不可として適用する(無ければ flex)。

### 1. 既存の手がかりを収集する

次を順に探して読む(存在するものだけ。全部無ければこの手順はスキップ):

- `tailwind.config.{js,ts,cjs}`、`theme.*`、`tokens.*`、CSS 内のカスタムプロパティ定義、SCSS 変数
- 代表的コンポーネント(ボタン・カード・入力欄)のスタイル
- ロゴ・ファビコンなどブランドカラーが分かるアセット、README、デザイン関連ドキュメント
- `AGENTS.md` / `GEMINI.md` のデザイン規約

見つけた値は「出典ファイル → 値」の形でメモしておく(手順7の報告で使う)。

### 2. ビジュアル参照(スクリーンショット・スライド・PDF)を読み取る

`designs/references/` 配下、およびユーザーが指定したパスの画像・PDF を開き、次の手順で分析する:

- **カラー**:①ページ背景 ②カード等の面 ③本文文字 ④最も目立つ操作要素(ボタン等)の4箇所から色を読む。グラデーション・写真・要素の境界付近は避け、**平坦な領域の中央**からサンプリングする。圧縮やアンチエイリアスで色がぶれるため、複数箇所から読んで丸める。
- **タイポグラフィ**:分類(セリフ/サンセリフ/幾何学系/ヒューマニスト)を判定し、近い実在フォントの候補を2つ挙げる。見出しと本文のウェイト差、おおよそのサイズ比も読む。
- **形**:角丸(なし / 小 4–6px / 中 8–12px / 大 16px+)、影の有無と強さ、区切り方(罫線か面か)、余白の密度(詰まった/標準/ゆったり)。

画像からの推定値はすべて**「推定」とマーク**し、報告時に仮定として列挙する。Google Slides が Drive 連携(MCP)経由で直接読める環境ならそこから取得してよい。読めない場合は「PNG/PDF に書き出して `designs/references/` に置いてほしい」とユーザーに依頼する。

### 3. トークンに正規化する

**矛盾の解決規則(この順で機械的に適用する)**:

1. ユーザーの明示指定 > コードから抽出した値 > 画像からの推定値
2. コード内で複数の値が競合 → 使用回数が多い方を正とし、報告に矛盾点として列挙する
3. どこからも取れない → 手順4の既定値を使い、報告に「既定値を採用」と明記する

**ニュートラル階調の作り方**:アクセント色の色相を 2〜5% 混ぜたグレーで、背景 → 面 → 罫線 → 補助文字 → 本文の5段を作る。純グレー(R=G=B)は使わない。

**ダークテーマの導出**(参照がライトのみの場合):明度の単純反転ではなく再設計する。背景は真っ黒ではなく #141414〜#1c1c1c 帯の有彩色寄り、本文は #e8e8e8〜#f0f0f0 帯、アクセントは彩度を保って明度 +10〜20%。コントラスト 4.5:1 を再確認する。

### 4. `designs/tokens.css` を生成する

次のスケルトンの**全トークンを埋める**。名前は変更・省略しない(以降のワークフローがこの名前を前提に動く)。ブランド固有のトークンを足すのは自由。

```css
:root {
  /* カラー: 面 */
  --bg: #___;             /* ページ背景 */
  --surface: #___;        /* カード・パネル */
  --surface-raised: #___; /* 浮いた面(メニュー・ポップオーバー) */
  --line: #___;           /* 罫線・区切り */
  /* カラー: 文字 */
  --ink: #___;            /* 本文 */
  --ink-muted: #___;      /* 補助文字 */
  /* カラー: アクセント */
  --accent: #___;
  --accent-ink: #___;     /* アクセント面の上に載る文字色 */
  /* カラー: セマンティック(アクセントとは別枠) */
  --ok: #___; --warn: #___; --danger: #___;
  /* タイポグラフィ */
  --font-display: /* 見出し用スタック */;
  --font-body: /* 本文用スタック */;
  --font-mono: /* 数字・コード用スタック */;
  --text-xs: ___; --text-sm: ___; --text-base: 1rem; --text-lg: ___;
  --text-xl: ___; --text-2xl: ___; --text-3xl: ___; /* 決めた倍率から算出 */
  --leading-body: 1.6; --leading-tight: 1.2;
  /* 余白・形 */
  --space-1: 4px; --space-2: 8px; --space-3: 12px; --space-4: 16px;
  --space-6: 24px; --space-8: 32px; --space-12: 48px; --space-16: 64px;
  --radius-sm: ___; --radius-md: ___; --radius-lg: ___;
  --shadow-sm: ___; --shadow-md: ___;
}
@media (prefers-color-scheme: dark) { :root { /* カラートークンのみ再定義 */ } }
:root[data-theme="dark"]  { /* 同じダーク値を再宣言(手動トグル用) */ }
:root[data-theme="light"] { /* ライト値を再宣言(手動トグル用) */ }
```

**既定値(どこからも取れなかった場合のみ)**:余白は上記スケルトンの値のまま、タイプスケールは 1.25 倍率、radius-sm/md/lg = 4/8/16px、フォントはシステムスタック(見出しと本文でスタックを分け、役割差はウェイトとサイズで作る)。

### 5. `designs/design-system.md` を生成する

次の見出し構成に従う:

```markdown
# <プロジェクト名> Design System
## 出典と確度      … どこから抽出したか(コード/画像=推定/既定値)を表で
## カラー          … 表: トークン名 / ライト値 / ダーク値 / 用途 / 出典
## タイポグラフィ  … 書体の役割と選定意図、タイプスケール表(名前/サイズ/用途)
## 余白・形        … 余白スケール、角丸・影の使い分け基準
## コンポーネント基本形 … ボタン(primary/secondary)・入力欄・カードの CSS
## Do / Don't      … このブランドで「らしい/らしくない」を具体例で各3つ以上
## 未決事項        … 仮定のまま残っている判断
```

### 6. スタイルガイドで検証する

`designs/design-system/index.html`(自己完結、tokens.css の内容をインライン)を生成する。含めるもの:カラースウォッチ(トークン名+値の表示付き)、タイプスケール見本、余白スケール、ボタン/入力欄/カードの実物、テーマ切替ボタン(`data-theme` をトグル)。

ブラウザで開き、**ライト・ダーク両方**のスクリーンショットを撮って確認する:

- [ ] 本文(--ink)/背景(--bg)のコントラストが両テーマで 4.5:1 以上
- [ ] アクセント面上の文字(--accent-ink)が両テーマで読める
- [ ] ダークテーマが「ライトの単純反転」になっていない
- [ ] ビジュアル参照がある場合、参照画像とスタイルガイドを並べて印象(色相・密度・角丸)が一致する

不一致があれば tokens.css を修正して再確認する。

### 7. 報告する

- 生成した3ファイルのパス
- トークンの出典表(コード由来 / 画像由来=推定 / 既定値)
- 矛盾していた値と採用の判断
- 以降の `/design` `/design-slides` はこのシステムを自動で使うことを伝える
