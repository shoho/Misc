# /design-iterate — デザインを対話で磨く

`/design` で作ったプロトタイプ(または指定された HTML)を、ユーザーのフィードバックに沿って改良する。

## 手順

### 1. 対象を特定する

ユーザーが指定しなければ `designs/` 配下で最後に更新されたデザインを対象にする。対象ファイルを読み、**変更前のスクリーンショット**を先に撮っておく(ビフォー/アフター比較に使う)。

`designs/.design-mode` が `strict` なら `.agent/rules/design-strict.md` を読んで適用する(下の翻訳表が「目安」から「既定の手順」になる)。

### 2. フィードバックを具体的な変更に翻訳する

曖昧な要望は、着手前に「どのプロパティをどう動かすか」に翻訳する。翻訳の目安:

| 要望 | まず疑うパラメータ |
|---|---|
| 重い・ごちゃごちゃする | 余白不足(space を1段階上げる)、罫線/影の過多(半分に減らす)、太字の多用 |
| 軽すぎる・寂しい | 階層不足(見出しと本文のサイズ差を拡大)、面の区切り不足、アクセント色の露出不足 |
| 今風に・モダンに | 角丸とタイプスケールの見直し、罫線区切り→余白区切り、彩度の調整 |
| ここを目立たせて | サイズ拡大より先に、コントラスト強化と周囲の余白を増やす(周囲を静かに) |
| 読みにくい | 行長 65ch 前後か、行間 1.5 以上か、コントラスト 4.5:1 以上か、を順に確認 |

複数の解釈がありうる場合は、採用した解釈を報告に明記する(確認待ちで作業は止めない)。

### 3. トークン単位で変更する

- 色・余白・タイプスケールの変更は `:root` のカスタムプロパティを更新して全体に波及させる。個別要素へのその場しのぎの上書き(`!important`、深いセレクタ)でカスケードを汚さない。
- ライト側の色トークンを変えたら、**対応するダーク側(`@media` と `[data-theme]` の全ブロック)も必ず更新**して整合を保つ。

### 4. 変更しない部分を守る

フィードバックに含まれない部分のデザイン判断は維持する。ファイルを作り直さず、既存ファイルへの最小の編集で実現する。

### 5. ブラウザで再検証する

変更後にスクリーンショットを撮り、ビフォー/アフターを並べて次を確認してから報告する:

1. 要望が意図どおり反映されたか
2. 無関係な部分が変わっていないか
3. 両テーマの整合が保たれているか

### 6. 報告する

「何を・なぜ・どう変えたか」を箇条書きで簡潔に。次に調整できそうな点があれば1〜2個だけ提案する。

## 補足

- 大きな方向転換(トーン自体の変更、レイアウト刷新)を求められたら、実装前に新しいデザインプランを短く提示してから進める。
- 反復の途中でユーザー指定と過去の自分の判断が矛盾したら、常にユーザー指定を優先する。
