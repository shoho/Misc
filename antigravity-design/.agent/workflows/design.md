# /design — プロンプトからプロトタイプを作る

ユーザーの説明から、ブラウザでそのまま開ける本物のプロトタイプ(HTML/CSS/JS)を作る。画像モックではなく、クリックできる実物を納品する。

## 手順

### 1. 題材を固定する

ユーザーの指示から次の3点を1行ずつ書き出す:

- **対象**(1つ): 何のページ/画面か
- **想定読者**: 誰が見るか
- **唯一の仕事**: このページが達成すべきこと1つ

不明瞭な点は質問で止めず、妥当な仮定を置いて作り、仮定を最後に明記する。

### 2. 既存デザインシステムを確認する

`designs/design-system.md` と `designs/tokens.css` が**存在すれば必ず読み**、色・書体・トークンをそこから使う。この場合、手順3のプランは「システムのどのトークンをどう使うか」の宣言になる — 新しい色や書体を発明しない。存在しなければスキル `frontend-design` の指針に従って自分で選ぶ。

### 3. デザインプランを提示する(コードより先に)

```
Color: <役割名>: #hex を4〜6色(背景/文字/アクセント/セマンティック)
Type: 見出し=<スタック>(理由1行)/ 本文=<スタック> / 必要なら数字・キャプション用
Layout: コンセプトを1〜2文
```

提示したら自己チェック:**題材名を伏せても「この題材のためのデザイン」と分かるか?**

- ❌ 汎用の例:「白背景、青アクセント、サンセリフ、カードグリッド」— どんな題材にも言えるので失格
- ✅ 固有の例:「家計簿:帳簿の罫線文化 → 面ではなく細罫線で区切る。数字が主役 → tabular-nums の大きな数字。インクは帳簿の藍色」

汎用に見える部分を修正してから実装に進む。

### 4. 実装する

- 出力先: `designs/<スラッグ>/index.html`(スラッグは英小文字ケバブケース。例: `kakeibo-dashboard`)
- **完全に自己完結させる**:CSS/JS はインライン、外部 CDN・外部フォント URL・外部画像に依存しない
- 必須要件:
  - [ ] `<html lang>` と `<meta name="viewport" content="width=device-width, initial-scale=1">`
  - [ ] 色は `:root` のカスタムプロパティ + 両テーマ4ブロック(`@media (prefers-color-scheme: dark)` / `:root[data-theme="dark"]` / `:root[data-theme="light"]`)。トークン定義部以外に生の HEX を書かない
  - [ ] 実データ・実コピーで作る(lorem 禁止)
  - [ ] レイアウトは flex/grid + gap。`:focus-visible` の可視化、`prefers-reduced-motion` の尊重
- デザイン原則ルールとスキル `frontend-design` を遵守する

### 5. ブラウザで検証する

Antigravity のブラウザツールで `index.html` を開き、スクリーンショットを撮って次の順で自分でレビューする:

1. 要素の重なり・はみ出し・ページ横スクロールが無いか
2. 意図した書体が効いているか(フォールバックで別物になっていないか)
3. 余白・色がカスケード衝突で打ち消されていないか
4. コンソールで `document.documentElement.dataset.theme = 'dark'` を実行してダークテーマも確認
5. ウィンドウ幅を狭めて(375px 相当)崩れないか

問題があれば修正して再確認する。スクリーンショットを成果物(Artifacts)として残す。

### 6. 報告する

ファイルパス、デザインプラン(確定版)、置いた仮定、次の反復の提案(「/design-iterate で調整できます」)を簡潔にまとめる。

## 禁止事項

- ワイヤーフレーム画像や説明だけで終わらせない。必ず動く HTML を作る。
- ユーザーが明示した色・トーン・参照を勝手に変えない。
- `designs/tokens.css` が存在するのに独自パレットを発明しない。
