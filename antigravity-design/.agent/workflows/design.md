# /design — プロンプトからプロトタイプを作る

ユーザーの説明から、ブラウザでそのまま開ける本物のプロトタイプ(HTML/CSS/JS)を作る。画像モックではなく、クリックできる実物を納品する。

## 手順

1. **題材を固定する**
   ユーザーの指示から「対象(1つ)・想定読者・そのページの唯一の仕事」を特定する。不明瞭な点は質問で止めず、妥当な仮定を置いて作り、仮定を最後に明記する。

2. **既存デザインシステムを確認する**
   `designs/design-system.md` と `designs/tokens.css` が存在すれば必ず読み、色・書体・トークンをそこから使う(`/design-system` で生成されるファイル)。無ければスキル `frontend-design` の指針に従って自分で選ぶ。

3. **デザインプランを提示する**(コードより先に)
   - Color: 名前付き HEX 4〜6色
   - Type: 書体ペア(見出し/本文、必要なら数字・キャプション用)
   - Layout: コンセプトを1〜2文
   汎用テンプレに見える部分があれば修正してから進む。

4. **実装する**
   - 出力先: `designs/<スラッグ>/index.html`(スラッグは英小文字ケバブケース)
   - 完全に自己完結させる:CSS/JS はインライン、外部 CDN・外部フォント URL・外部画像に依存しない
   - カラーパレットは `:root` のカスタムプロパティとして定義し、ライト/ダーク両テーマをトークン差し替えで実装する
   - 実データ・実コピーで作る(lorem 禁止)
   - デザイン原則ルールとスキル `frontend-design` を遵守する

5. **ブラウザで検証する**
   Antigravity のブラウザツールで `index.html` を開き、スクリーンショットを撮って自分でレビューする:
   - 要素の重なり・はみ出し・横スクロールが無いか
   - フォントフォールバック・カスケード衝突で意図と違う見た目になっていないか
   - 可能ならダークテーマ(`data-theme="dark"` を付与)も確認する
   問題があれば修正して再確認する。スクリーンショットを成果物(Artifacts)として残す。

6. **報告する**
   ファイルパス、デザインプラン(確定版)、置いた仮定、次の反復の提案(「/design-iterate で調整できます」)を簡潔にまとめる。

## 禁止事項

- ワイヤーフレーム画像や説明だけで終わらせない。必ず動く HTML を作る。
- ユーザーが明示した色・トーン・参照を勝手に変えない。
