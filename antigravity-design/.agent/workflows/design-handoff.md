# /design-handoff — 実装へのハンドオフ

`designs/<スラッグ>/` のプロトタイプを、実装エージェント(Claude Code・Antigravity 自身・人間の開発者)にそのまま渡せる仕様一式にまとめる。

## 手順

### 1. 対象を特定する

指定が無ければ `designs/` 配下で最後に更新されたデザインを対象にする。プロトタイプ HTML を読み直し、実際に使われているトークン・状態・インタラクションを洗い出してから書く(記憶で書かない)。

### 2. `designs/<スラッグ>/HANDOFF.md` を生成する

次のテンプレート構成に従う。「プロトタイプを見れば分かる」で済ませず、値と理由を本文に書き出す:

```markdown
# <デザイン名> — 実装ハンドオフ

## 概要
何のための画面/ページか、対象ユーザー、達成すべき仕事(1段落)

## デザイントークン
このデザインで使用した色・書体・スケールの表(トークン名 / ライト値 / ダーク値 / 用途)。
`designs/tokens.css` があればそこへの参照+このデザイン固有の差分だけを列挙。

## レイアウト構造
セクション/コンポーネントの分解(箇条書きの階層で)。それぞれの責務と、
ブレークポイントごとのレスポンシブ挙動(何が折り返し、何が非表示になるか)。

## インタラクション仕様
hover / focus / active / disabled の各状態、状態遷移、
アニメーション(プロパティ・duration・easing)、`prefers-reduced-motion` 時の挙動。

## 両テーマ
ライト/ダークで値が変わるトークンと、テーマ固有の注意点(影の扱い等)。

## アクセシビリティ要件
フォーカス順、コントラスト実測値、キーボード操作、ARIA が必要な箇所。

## 実装ノート
プロトタイプ HTML のどこを見れば各部の実装が分かるか(行番号 or セクション名)。
プロトタイプと本実装で変えるべき点(データ結合、コンポーネント分割の提案、
プロトタイプ用の簡易実装で本番では置き換えるべき箇所)。

## 未決事項
仮定のまま残っている判断(ユーザーに確認が必要なもの)
```

### 3. アセットを揃える

最終スクリーンショット(両テーマ)を `designs/<スラッグ>/` に保存し、HANDOFF.md から相対パスで参照する。

### 4. 報告する

バンドルの場所と、実装エージェントに渡すときのひとこと指示例(例:「designs/<スラッグ>/HANDOFF.md を読んで、この仕様どおりに実装して」)を示す。
